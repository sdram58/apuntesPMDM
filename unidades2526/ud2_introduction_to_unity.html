Introducción

¿Qué es Unity?

Unity es un motor de videojuegos multiplataforma.

Su origen se remonta a 2005.

Permite el desarrollo de juegos 2D y 3D.

Actualmente, Unity admite desarrollo para alrededor de 20 plataformas, siendo las más populares PC, iOS y Android.

Unity ofrece un conjunto de herramientas tan completo que minimiza la necesidad de herramientas externas para completar proyectos.

Unity Hub

El programa Unity Hub permite gestionar las versiones del editor de Unity que quieras utilizar.




Para usar Unity Hub, necesitas disponer de un Unity ID


También puedes crear el Unity ID desde la web de Unity: https://id.unity.com/es/



Con un Unity ID, además de poder usar el motor (licencia de uso), puedes acceder a la Unity Asset Store, donde podrás descargar assets gratuitos y de pago.

https://assetstore.unity.com/

  • Añade módulos de destino según plataforma (Android/iOS/Windows/Linux/WebGL/Consolas). Normalemente, Windows o Linux
  • Para proyectos mobile: marca Android Build Support (SDK & NDK) y/o iOS.

Instalación de Unity

Para desarrollar videojuegos, necesitas el editor de Unity.

La primera vez que inicias sesión en Unity Hub con tu Unity ID, se te invitará a instalar un editor de Unity.

Por defecto, se propone instalar la versión más reciente, no es LTS (Long Term Support), pero es la recomendada.

Para este curso instalaremos la versión más reciente 6.2.


Marcamos las opciones a instalar para las plataformas que queramos trabajar



Aceptamos las condiciones tras leerlas.


Debes elegir la configuración de módulos para crear ejecutables por plataforma, añadir idiomas y documentación de Unity.

Módulos recomendados

  • Windows/Mac/Linux Build Support (IL2CPP).
  • Android Build Support (SDK, NDK, OpenJDK) y/o iOS Build Support.
  • WebGL Build Support si se requiere web.
  • Plantillas con URP para proyectos 2D/3D ligeros.

Para clase, si usas Linux, suele bastar con seleccionar Linux Build Support (IL2CPP).


Editor de código

Unity requiere un editor para scripting; en Windows, la instalación incluye Microsoft Visual Studio Community. En clase, quien lo tenga detectará Visual Studio Code y podrá establecerlo como editor por defecto.

Si no quieres instalar ese editor, desmárcalo.

Si usas Microsoft Visual Studio Community o Visual Studio Code como editor por defecto, instala las extensiones para C#, el lenguaje utilizado con Unity.



Si no tienes Microsoft Visual Studio Community y no deseas instalarlo, desmárcalo.

Puedes usar cualquier otro editor cambiando la configuración del editor de Unity una vez instalado, en Menu → Edit → Preferences.



Si lo necesitas más adelante, puedes añadir o quitar módulos al editor.

Unity Hub será el punto de partida de cualquier proyecto. En su ventana, verás las secciones:

  • Projects: gestor de proyectos.

Editor de Unity

Por defecto, la ventana del editor de Unity se muestra así:


Unity utiliza Programación Orientada a Objetos, por lo que todos los elementos del juego se incluyen en la jerarquía de objetos.

El juego ocupa la posición más alta de la jerarquía y, dentro de él, están las scenes, que pueden considerarse las pantallas del juego.

La hierarchy muestra todos los elementos de una scene.


En cada scene debes añadir todos los elementos necesarios; estos elementos son GameObjects. Las scenes también siguen una estructura jerárquica, permitiendo agrupar GameObjects entre sí.

Por defecto, un proyecto de Unity contiene una Scene con un GameObject de tipo camera llamado Main Camera, un GameObject de tipo light llamado Directional Light y un GameObject para aplicar efectos de postprocesado Global Volume.

En la parte central del editor puedes ver la scene view que, al pulsar el botón de reproducción, pasará a ser la vista desde la cámara configurada en el juego.


Al seleccionar un objeto de la scene, el Inspector muestra todas sus características.


Desde el código de scripting puedes acceder a los componentes de cada objeto.

Cada tipo de objeto tendrá componentes por defecto, y se pueden añadir más si es necesario.



El File Manager permite ver y realizar acciones sobre los archivos del proyecto.

Se recomienda crear desde el administrador las carpetas y archivos que necesites.


La documentación oficial detalla la interfaz de Unity: https://docs.unity3d.com/Manual/unity-editor.html


Consejo de productividad

En Project Settings → Editor → Enter Play Mode Settings, puedes desactivar Domain Reload/Scene Reload para acelerar la entrada en Play Mode (ten en cuenta efectos colaterales y reinicializa estados manualmente cuando proceda).

Ejecución del juego

Cuando el juego está en ejecución (botón "Play" ▶ en la barra superior), la vista cambia a la cámara y la interfaz se oscurece.


Durante la ejecución puedes modificar valores para ver reacciones, pero los cambios solo persisten mientras el juego está en ejecución.

Como hay poca diferencia visual entre estados, a veces se olvida que el juego está corriendo y se pierden cambios.


Ten en cuenta que en ejecución puedes cambiar parámetros, pero al volver al modo edición dichos parámetros recuperarán su estado original. Verifica siempre en qué modo estás.

Desde Menu → Edit → Preferences, en la sección Colors, puedes decidir el color de superposición cuando el juego está en ejecución: Playmode tint.

Es útil elegir un color de alto contraste para distinguir fácilmente si el juego está corriendo o no.


Una vez cambiado, será más fácil ver que el juego está en ejecución.



Añadiendo código ejecutable

Para añadir código ejecutable al proyecto, debes crear archivos que lo contengan; estos archivos se denominan scripts.

Hay varias formas de añadir scripts, pero lo más apropiado es hacerlo desde el administrador de archivos.

Además, conviene mantener una buena organización: es habitual crear la carpeta Scripts dentro de Assets y ubicar allí todos los scripts necesarios.


Con la carpeta Scripts creada, haz clic derecho sobre ella o sobre el icono de carpeta y selecciona Create → MonoBehaviour Script


También lo podemos hacer desde la opción create → script → MonoBehaviour Script. Cabe destacar que hay más opciones de creación de Script, pero inicialmente usaremos la MonoBehaviour (clase de la que hereda que contiene funciones del ciclo de vida)



En este punto, escribe el nombre del archivo siguiendo las convenciones de C# similares a Java (CamelCase con la primera letra en mayúscula).

Además, el script debe tener un nombre que identifique claramente lo que hace.

Al seleccionar un Script en el File Manager verás su código en el Inspector, pero no se puede editar desde ahí.


Haciendo doble clic sobre el Script, se abrirá en el editor de código configurado.



Estructura por defecto de un Script de Unity

Un Script por defecto de Unity tiene la siguiente estructura:


La frecuencia del game loop puede variar, por lo que el método Update no se ejecuta siempre el mismo número de veces.

El siguiente código mostrará mensajes en la consola cuando se ejecute el script.

using System.Collections;
using System.Collections.Generic;
using UnityEngine;

public class HelloThere : MonoBehaviour
{
    // Start is called before the first frame update
    void Start()
    {
        Debug.Log("Start --> Hello there!");
    }

    // Update is called once per frame
    void Update()
    {
        Debug.Log("Update --> Hello there!");
    }
}


Para probarlo, pulsa el botón "Play" ▶ en la parte superior del editor.


Como se ve, no aparecen mensajes en la pestaña Console.



Esto se debe a que para ejecutar un script, debe estar adjunto a un GameObject.

Puedes adjuntar un script a un GameObject arrastrándolo desde el administrador al GameObject de varias formas:

  • Arrastrando el script sobre el GameObject en la jerarquía.
  • Arrastrando el script sobre el GameObject en la vista de la scene.
  • Seleccionando el GameObject y arrastrando el script al Inspector.

Una vez adjunto, el script se convierte en un componente del GameObject y aparece en el Inspector.


Ahora, al ejecutar el juego, el script se ejecutará.


Es importante marcar la opción Collapse, que consolida mensajes iguales en una sola línea e incrementa el recuento.

También, activando Stats en la vista, puedes ver información como los FPS (frames por segundo).

Scripting

Como en cualquier aplicación software, los juegos necesitan que su funcionalidad sea programada.

En Unity, la funcionalidad se programa mediante scripts.

Con scripts puedes:

  • Responder a entradas del jugador.
  • Ejecutar eventos en el momento adecuado.
  • Crear efectos gráficos.
  • Controlar el comportamiento físico de GameObjects.
  • Implementar un sistema de IA para NPCs del juego.
  • …

Actualmente, el único lenguaje recomendado para crear proyectos con Unity es C#.

UnityScript (derivado de JavaScript) y Boo (similar a Python) forman parte del legado y ya no se usan ni se recomiendan en proyectos modernos.

C# se creó en 2000 para ser compatible con la plataforma .NET de Microsoft.

Hereda buenas características de C++, Java y Visual Basic.

Es un lenguaje nativo orientado a objetos.

Gracias a la librería IL2CPP, los scripts creados en Unity se convierten a C++ para compilar nativamente en la plataforma de destino.

Un script de Unity tiene la siguiente estructura básica:

  • UnityEngine: importa la clase MonoBehaviour para definir scripts asociados a GameObjects.
  • System.Collections: librería .NET con listas, arrays, tablas hash, etc.
  • Clase definida que extiende de MonoBehaviour:
    • Método Start: se usa para inicializar el GameObject.
    • Método Update: se ejecuta una vez por frame (game loop).

En C#, todas las instrucciones deben terminar con ; igual que en Java.

Variables

Las variables en C# se definen igual que en Java:

int age;

También pueden inicializarse en la declaración:

int age = 21;

Una vez declarada, su valor puede cambiarse en cualquier momento:

age = 22;

Si las variables se declaran como propiedades de clase, su acceso puede establecerse como public o private.

Si una variable se declara dentro de un método, no debe indicarse el tipo de acceso; solo será visible en ese método.


Las variables declaradas como propiedades public serán visibles en el Inspector durante la ejecución, permitiendo cambiar su valor desde ahí.



Recuerda que los cambios hechos durante la ejecución son temporales: al detener, se vuelve al estado inicial.

Métodos

Como en cualquier lenguaje orientado a objetos, puedes crear métodos dentro de las clases.

En C#, para crear un método, sigue esta estructura:

  • Modificador de acceso (si no se especifica, por defecto es private).
  • Tipo de retorno (void si no devuelve nada).
  • Nombre del método (primera letra en mayúscula y CamelCase).
  • Paréntesis para los parámetros, si los hay.
  • Llaves para el bloque de código.

Clase MonoBehaviour

Al crear un script C# en Unity, verás que la clase hereda de MonoBehaviour.

La clase MonoBehaviour es necesaria para scripts asociados a GameObjects del proyecto.


Unity funciona por eventos, por lo que en clases que heredan de MonoBehaviour puedes incluir métodos predefinidos que se ejecutarán cuando ocurran los eventos asociados.

Por defecto, se añaden dos métodos que se ejecutan con eventos específicos:

  • Start: se llama en el frame en que se activa el script y justo antes de la primera llamada a cualquier Update.
  • Update: se llama en cada frame.

Hay muchos más métodos en la documentación: https://docs.unity3d.com/ScriptReference/MonoBehaviour.html

Game Loop

El game loop puede definirse como las distintas fases por las que pasa un script durante la ejecución:

  • Initialization
  • Physics Update
  • Input Update
  • Game Logic
  • Destruction

Cada fase se asocia con un conjunto de métodos de MonoBehaviour:


Al ejecutar el juego, Unity inicia el game loop y, en cada fase, llamará a los métodos correspondientes que se encuentren en los scripts añadidos a los GameObjects de la scene.

Ejemplo de script implementando algunos métodos de MonoBehaviour :

using System.Collections;
using System.Collections.Generic;
using UnityEngine;

public class SphereEvents : MonoBehaviour
{
    void Awake(){
        Debug.Log("Estoy en el Awake de la esfera");
    }

    
    void Start(){
        Debug.Log("Estoy en el Start de la esfera");
    }
    // Update is called once per frame
    void Update(){
        Debug.Log("Estoy en el Update de la esfera");
    }

    void OnEnable(){
        Debug.Log("Estoy en el OnEnable de la esfera");
    }

    void OnDisable(){
        Debug.Log("Estoy en el OnDisable de la esfera");
    }
}
