Types of Applications

The mobile phone market is extremely diverse. Every month, dozens of new smartphones are launched, and as a result, mobile developers and entrepreneurs face challenges due to the increasing number of mobile applications. There is no magic solution for choosing the platform to use.

Both native mobile applications and cross-platform apps have their advantages and disadvantages. As a business owner, you should understand the differences between them.

Advantages of Native Application Development

  • You have access to the full potential of the mobile device

    The essence of native apps is that they are written for a specific platform (today, Android or iOS). This means developers can provide much better integration between app functionalities and the mobile hardware features: Bluetooth, NFC, GPS sensor, camera, microphone, accelerometer, gyroscope, etc.

  • Better screen performance

    Using a native language gives you more freedom in terms of screen optimization. With the number of mobile devices available and their varying screen sizes and aspect ratios, it’s crucial to ensure your apps display correctly everywhere.

  • Better performance

    Along with control and flexibility, better integration with mobile hardware provides another significant advantage. Native apps are very fast and perform much better than others, like web apps. Native apps maximize the use of processing power, minimizing response time as much as possible.

  • Internet connection is not mandatory.

    Since native apps are installed directly on the mobile device, they can access app data offline. On the other hand, a web app or any other platform requires a constant online connection, which depends on the network speed.

    Combined with the performance mentioned in the previous point, the ability to work offline makes native apps the perfect choice for developing apps like mobile games.

Disadvantages of Native Applications

If there’s any disadvantage to native app development, it’s the incompatibility with other mobile operating systems. iOS native apps are developed using Swift or Objective-C and can only be installed on iPhone or iPad. Similarly, Android apps are not compatible with iOS because they are developed in Java or Kotlin.

Another aspect worth mentioning is version compatibility. For example, due to the wide variety of Android versions, providing full support to all app users becomes a challenge.

Advantages of Cross-Platform Application Development

  • Reusable code

    Instead of developing different apps, mobile developers can use the same code for all platforms. This means, unlike a native app, with cross-platform development, you can implement a feature in an app for one operating system, and it will work on others.

    This reusability also offers an advantage when it comes to app deployment.

  • Cost-effectiveness

    Implementing your idea as a cross-platform app does not require large investments in resources. This is because you don’t need a dedicated team for each mobile operating system. Thanks to a short learning curve and a unified set of technologies, a single team can support all platforms.

  • The same UI and UX across all platforms.

    App performance is undoubtedly important, but equally important is how the app looks and flows. It can be difficult to achieve the same user interface and user experience simply because two independent systems are involved.

    Using a single codebase allows cross-platform apps to maintain the same appearance and user experience on all mobile devices.

Disadvantages of Cross-Platform Application Development

  • They are not as flexible as native apps.
  • Performance is not their strongest point.
  • The process for adding them to app stores can be more complicated.


Android Features

While there are several mobile platforms, the Android operating system for mobile devices offers a set of features that differentiate it from others:

  • Open platform. It is a free development platform based on Linux and open source. Among its advantages is the ability to use and adapt the system without paying royalties.
  • Guaranteed portability. Thanks to the virtual machine concept, final applications are developed in Java, ensuring they run and function on a wide variety of devices, both current and future.
  • Architecture based on Internet-inspired components. For example, the user interface design is done in XML, allowing the same app to run on a mobile phone with a small screen or on a Netbook.
  • Philosophy of a device always connected to the Internet.
  • Many integrated services. For example, GPS-based location, SQL databases, voice recognition and synthesis, browser, maps, etc.
  • Acceptable level of security. Programs are isolated from each other thanks to the concept of execution within a sandbox provided by the virtual machine.
  • Optimized for low power consumption and memory usage. Android uses the Dalvik virtual machine, Google's implementation of the Java virtual machine optimized for mobile devices.
  • High-quality graphics and sound. Smooth vector graphics, Flash-inspired animations, 3D graphics based on OpenGL, and standard audio and video codecs.

Android is an operating system aimed at mobile devices. The company, like many other tech companies, was founded in Palo Alto, California, in 2003. However, due to its rapid growth and potential, it caught Google's attention, and they purchased it in 2005 for 50 million dollars.



Android Architecture

Android implements an operating system based on Linux, and its virtual machine (Dalvik) compiles Java files into .apk applications. An apk (*) contains Java classes compiled into a bytecode format known as DEX. The DEX format is processor architecture-independent, so to be executed, it must be translated into native "machine code" for the processor in our device. Naturally, this is slower compared to direct execution.
*We now have AAB (Android App Bundle) instead.

To solve this issue, Google introduced a new virtual machine called ART starting with KitKat version 4.4.

The fundamental difference between Dalvik and ART is that Dalvik uses just-in-time (JIT) compilation, while ART uses ahead-of-time (AOT) compilation. Dalvik executes a virtual machine that interprets code as the application runs, whereas ART performs pre-compilation during the app installation process.

Among other differences, ART also offers improvements in garbage collection, as well as in application development and debugging.


As can be seen, Android internally makes use of numerous libraries, such as SGL, OpenGL, OpenCORE, WebKit, SSL, FreeType, SQLite, and libc.

Nowadays, Android is present in all kinds of devices, primarily in smartphones, tablets, smartwatches, Android TV, Android Auto, etc. Even with the appearance of the Android-x86 project, it is possible to use the open-source Android operating system installed on a PC, such as the adapted version of Android 9 Pie.

One of the main challenges Android developers face is the vast number of different devices, screen formats, and common usage orientations, as well as varying screen densities. Indeed, when designing an app, we must decide which types of devices we want it to function smoothly on and how many different screen layouts we will need to design.

Another significant challenge is fragmentation: there are too many versions coexisting in the market, each with its own peculiarities, and they are released so quickly that it’s hard to keep up. Therefore, when designing an app, we must choose from which version it will work. However, there are support libraries available to use older features on modern devices.

One little-known peculiarity by non-developers is that in the Android world, the OS version does not always align with the API we must use to create an app for it. We will explore this relationship shortly.

Fragmentation

Android fragmentation refers to the various versions of the same operating system. As with other operating systems, Android evolves by adding new security improvements, new features, and to differentiate them, different versions are created.

In Android, we must distinguish between the version number and the API level. When developing applications, we need to consider the selected API level to see which versions or devices can run our application. Although developers have created libraries to maintain backward compatibility with previous versions.

Android Studio

Introduction

Android Studio (AS) is Google’s official IDE for creating applications that run on its popular operating system. It replaced the former official Android environment, Eclipse (https://www.eclipse.org/), and allows developers to choose between developing apps with Java or Kotlin (https://kotlinlang.org/).

Unlike AppInventor (http://appinventor.mit.edu/), which allows for easy app generation via block-based graphical programming, with Android Studio, you have to work from scratch. Indeed, as we will see, working with Android Studio is laborious, but the benefit is having clear and almost total control over what you are building.

Applications created using PhoneGap (https://phonegap.com/) or its sibling Apache Cordova (https://cordova.apache.org/) are hybrid apps (using HTML / CSS / JS), whereas applications created with AS will be fully native, giving full access to system hardware resources.

The AVD Manager allows us to create devices tailored to our preferences and needs.

Testing on Real Devices

USB Driver Installation, Developer Mode, and USB Debugging

While Android Studio allows us to test our applications using the Android Virtual Device (AVD) emulator, it's evident that nothing beats testing them on our actual mobile device (whether it's a phone, tablet, etc.), or even on multiple devices of different sizes and/or Android versions. We can use drivers from the device manufacturer, Google's drivers, or the well-known universal ADB drivers. This doesn't stop us from further testing our programs on different devices, screens, versions, etc. The AVD emulator is also commonly used.

Before installing the driver, we need to put our Android device(s) in developer mode. This is crucial as it allows us to test our application and even debug it directly from AS with the mobile device connected via USB. To do this, go to Settings, then About Phone, and tap Build Number seven times. After that, a message will appear indicating that you are now a developer.


Returning to settings, a new menu called Developer Options should now appear, where we must enable USB Debugging.

Note: Pay attention to the image; USB debugging is disabled.

Now let's install the USB driver for the device. If we don't have it installed, the manufacturer might provide one, or it might install automatically via Internet updates, or even ask us to install it directly when connecting the phone to the computer.

Installing Google's drivers beforehand may help. Go to Tools -> SDK Manager -> SDK Tools and check the box for Google USB Driver. Then press OK twice and Finish.


In my case, I connected three different devices, and they worked without any action, as seen in the Device Manager. Modern devices usually don't have issues with Windows.
